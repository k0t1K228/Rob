# 📖 RobloxSkibidi TD — документация для разработчиков

Tower Defense на Roblox. Код живёт в файлах этого репозитория и синхронизируется
в Roblox Studio через **Rojo**. Карта (геометрия, модели) живёт в Studio.

> **Золотое правило:** код пишем ТОЛЬКО в VS Code. Скрипты в Studio не редактируем —
> при следующем подключении Rojo их перезапишет файловой версией.

---

## 1. Установка с нуля (новый разработчик)

### 1.1 Инструменты

1. Установи **Rokit** (менеджер тулчейна Roblox): https://github.com/rojo-rbx/rokit
   — скачай релиз, запусти `rokit self-install`.
2. В папке проекта выполни:
   ```powershell
   rokit install
   ```
   Он прочитает `aftman.toml` и поставит нужную версию Rojo (7.6.1).
3. Установи плагин Rojo в Roblox Studio:
   ```powershell
   rojo plugin install
   ```
4. В VS Code поставь расширения: **Rojo** и **Luau Language Server** (JohnnyMorganz).

### 1.2 Проверка, что всё работает

```powershell
cd <папка проекта>
rojo --version    # → Rojo 7.6.1
rojo build -o test.rbxlx   # → "Built project ..." без ошибок
```

## 2. Ежедневная работа

```powershell
cd <папка проекта>
rojo serve
```

→ Открой place в Roblox Studio → вкладка **Plugins** → иконка **Rojo** → **Connect**
(хост `localhost`, порт `34872`).

После подключения правки `.luau`-файлов мгновенно прилетают в Studio.
Тестируешь кнопкой **Play** в Studio.

**Типичные проблемы:**

| Симптом | Причина / решение |
|---|---|
| `Failed to find tool 'rojo' in any project manifest` | Пустой/битый `aftman.toml`. В нём должно быть `rojo = "rojo-rbx/rojo@7.6.1"` в секции `[tools]` |
| `error 10048 ... одно использование адреса сокета` | Порт 34872 уже занят другим `rojo serve`. Закрой старый терминал или `Get-Process rojo \| Stop-Process -Force` |
| Логика срабатывает дважды | В Studio остались СТАРЫЕ копии скриптов рядом с теми, что кладёт Rojo. Удали дубликаты из Studio |
| Странные сбои синхронизации | Проект лежит в OneDrive — облако может лочить файлы. Поставь синк OneDrive на паузу или перенеси проект |

## 5. Башни

### 5.1 Как работает

`GameManager` сканирует корень Workspace (на старте и через `ChildAdded`).
Модель считается башней, если её **атрибут `TowerType`** или **имя** совпадает
с ключом из `TOWER_CONFIGS` в `src/shared/Classes/TowerClass.luau`.

Конфиг — это «скин»: одна и та же модель камеры с разными конфигами становится
винтовкой, огнемётом или лазером.

### 5.2 Готовые конфиги

| Ключ | Цена | Оружие | Тип атаки | Урон | Скоростр. | Радиус | Особенности |
|---|---|---|---|---|---|---|---|
| `Rust` | 150 | винтовка | single | 25 | 1/с | 26 | классика, одна цель |
| `RustFlamer` | 200 | огнемёт | **cone** | 8 | 2/с | 18 | конус 70°, бьёт ВСЕХ в секторе |
| `RustLaser` | 250 | лазер | ramp | 6 | 2/с | 32 | +50%/сек на одной цели (макс ×5), непрерывный луч |
| `RustRocket` | 300 | ракеты | single | 40 | 0.4/с | 38 | **неуправляемая** ракета: летит в точку, где был враг при пуске, взрыв по площади (радиус 8); быстрые враги уворачиваются |
| `RustMelee` | 100 | ближний бой | multi | 18 | 1.2/с | 10 | бьёт всех вокруг себя |
| `RustSonic` | 200 | звуковая волна | multi | 10 | 0.7/с | 20 | волна по всем в радиусе |
| `RustHybrid` | 400 | огнемёт/ракеты | cone/multi | 8/15 | 2/0.4 | 18/34 | **два режима**: «Огнемёт» ⇄ «Залп ракет» (3 неуправляемые ракеты) |
| `RustConverter` | 350 | вербовка | **convert** | — | 0.5/с | 24 | шанс 35% сделать врага союзником; режимы «Турель» ⇄ «Возврат» |

**Магазин/перезарядка:** Винтовка 8 выстрелов / 2.5 с, Огнемёт 14 / 3 с,
Лазер 12 / 3 с, Ракетница 4 / 4 с. У Клинка и Звуковой перезарядки нет.
**HP башни:** у всех 100 (ракета босса-разрушителя = 25 → 4 попадания).

### 5.3 Поля конфига (справочник)

```lua
MyTower = {
    displayName = "Моя башня", -- имя в магазине и меню
    price      = 250,       -- цена в магазине
    damage     = 25,        -- урон за выстрел
    fireRate   = 1,         -- выстрелов в секунду
    range      = 20,        -- радиус атаки (studs)
    attackType = "single",  -- "single" | "multi" | "ramp" | "cone"
    weapon     = "rifle",   -- "rifle"|"flamethrower"|"laser"|"rocket"|"melee"|"sonic"
    beamColor  = Color3.fromRGB(255, 100, 0), -- цвет эффектов и циферок урона

    -- необязательные (есть значения по умолчанию):
    maxTargets        = 1,  -- для multi: сколько целей одновременно
    splashRadius      = 0,  -- для rocket: радиус взрыва. Ракета неуправляемая —
                            -- летит в точку, где был враг в момент выстрела
    rampPerSecond     = 0,  -- для ramp: прирост урона за секунду (0.5 = +50%)
    maxRampMultiplier = 1,  -- для ramp: потолок множителя
    coneAngle         = 60, -- для cone: угол конуса в градусах
    scale             = 1,  -- размер башни: 0.5 маленькая / 1 средняя / 2 большая
    maxHealth         = 100, -- HP башни (её сносят боссы и стан-враги)
    magazine          = 0,  -- выстрелов до перезарядки (0 = бесконечно)
    reloadTime        = 0,  -- длительность перезарядки, сек

    -- для attackType = "convert" (вербовщик):
    convertChance     = 0.35,     -- шанс вербовки за попытку
    convertMode       = "turret", -- "turret" (стоит и стреляет) | "return" (идёт в рукопашную)
    allyDamage        = 12,       -- статы, которые получит союзник
    allyFireRate      = 1,
    allyRange         = 18,

    -- СВИТЧ-МОДЫ: добавь massив modes — башня сможет переключать
    -- режимы прямо в игре через меню наведения. Каждый режим —
    -- оверрайд поверх базовых полей (что не указал — берётся из базы):
    modes = {
        {name = "Режим А", attackType = "cone",  weapon = "flamethrower", damage = 8,  fireRate = 2},
        {name = "Режим Б", attackType = "multi", weapon = "rocket", damage = 15, maxTargets = 3, splashRadius = 5},
    },
}
```

Чтобы в магазине появилась кнопка новой башни — добавь её ключ в `SHOP_ORDER`
(там же, в `TowerClass.luau`).

### 5.4 Как добавить новую башню/скин — пошагово

1. Открой `src/shared/Classes/TowerClass.luau`.
2. Добавь новый ключ в `TOWER_CONFIGS` (см. справочник выше).
3. В Studio: возьми модель башни, в **Properties → Attributes** добавь
   атрибут `TowerType` (string) = твой ключ. Либо просто назови модель этим ключом.
4. Поставь модель в корень Workspace (и внутрь зоны, если есть `PlacementZones`).
5. Запусти Play — в Output будет `Башня активирована: <ключ>`.

⚠️ Визуал оружия сейчас — placeholder из неоновых Part'ов. Реальные модели
и партиклы — задача 4 (Чурюмов). Точка крепления эффектов — `PrimaryPart` башни.

## 6. Враги

- HP: 100 (поле `maxHP` в `EnemyClass.new`).
- Движение: плавное через `Heartbeat`, скорость и прочее — таблица `CONFIG`
  вверху `EnemyClass.luau` (speed, rotateSpeed, waypointReach, baseReach, spread).
- У каждого врага случайный разброс ±4 studs от маршрута, чтобы толпа не шла строем.
- **Камикадзе:** дойдя до базы, враг наносит ОДИН удар, равный его **текущим** HP,
  взрывается и умирает. Подбитый враг бьёт слабее ⇒ дамажить врагов по пути выгодно,
  даже если не успеваешь добить.
- Шаблон клонируется из `ServerStorage["skibidi toilet"]`.
- **Стан-враги** (`stunners = N` в волне): помечены жёлтым шаром; каждые 6 с
  глушат ближайшую башню в радиусе 14 на 5 с (константы — `CONFIG` в EnemyClass).
- **Перевербовка** (башня «Вербовщик»): враг становится зелёным союзником
  навсегда (`enemy.friendly = true`, атрибут `Friendly`). Башни по нему не стреляют,
  волна его не ждёт, награды за его смерть нет. Режим «Турель» — стоит и стреляет
  по врагам; «Возврат» — догоняет ближайшего врага и дерётся врукопашную
  (враг отбивается на 8 урона/удар). Боссов вербовать нельзя.

⚠️ **PrimaryPart у шаблона обязателен** — модель двигается через pivot.

### 6.1 Боссы («вертолёты») — `BossClass.luau`

Боссы НЕ ходят по маршрутам: подлетают к башне и **кружат вокруг неё по орбите**
(нос по направлению полёта, как настоящий вертолёт на боевом заходе), атакуя
по таймеру; после серии атак перелетают к следующей башне. Приоритет цели —
башня с наибольшим **нанесённым уроном** (`tower.totalDamage`), с шансом 30%
летят к случайной. Спавнятся в начале волны (`bosses = {...}` в WAVES).

| Ключ | HP | Оружие | Что делает с башней | Атак до перелёта |
|---|---|---|---|---|
| `StunCopter` | 600 | лазер | глушит на 5 с (станер не долбит уже оглушённую) | 2 |
| `DestroyerCopter` | 900 | ракеты | −25 HP за попадание (4 попадания = взрыв башни) | 4 |

- Башни стреляют по боссам тем же кодом, что по врагам (интерфейс совместим).
- Награда за убитого босса: `Economy.BOSS_REWARD` (100₵ каждому).
- Новый босс = новый конфиг в `BOSS_CONFIGS`. Поля: `hp, speed, flyHeight,
  weapon ("laser"|"rocket"), attackRange, fireRate, stunDuration, damage,
  attacksBeforeMove, beamColor, bodyColor, templateName`.

#### Своя модель босса (вместо заглушки из Part'ов)

Положи модель в **ServerStorage** с именем **`Boss_<Тип>`**
(`Boss_StunCopter`, `Boss_DestroyerCopter`), либо укажи произвольное имя
в поле `templateName` конфига. За что отвечает каждая деталь модели:

| Деталь | Обязательно | За что отвечает |
|---|---|---|
| **PrimaryPart** | да | корпус: центр полёта и поворотов, хитбокс для башен, к нему клеится HP-бар и привариваются остальные детали |
| детали с `Rotor` в имени (`Rotor`, `TailRotor`, `MainRotor1`…) | нет | **крутятся автоматически** вокруг своей вертикальной оси, следуя за корпусом |
| деталь `Muzzle` (можно глубоко внутри модели) | нет | точка, откуда вылетают ракеты/лазер; нет — стреляет из корпуса |
| все остальные детали | — | просто визуал: привариваются к PrimaryPart и летят вместе с ним |

Что происходит автоматически при спавне: Humanoid удаляется, все детали
(кроме роторов) привариваются WeldConstraint'ами к PrimaryPart,
`CanCollide/CanQuery = false`. Двигается только PrimaryPart —
модель не может развалиться или отстать от хитбокса.

## 7. Маршруты и события карты

### 7.1 Формат WaypointsFolder

Два варианта:

```
WaypointsFolder/            WaypointsFolder/
├── 1   (Part)              ├── Route1/  (Folder)
├── 2   (Part)              │   ├── 1, 2, 3...  (Part'ы)
└── 3   (Part)              ├── Route2/
                            │   └── 1, 2, 3...
один маршрут "Main"         └── Route3/ ...
                            каждый враг выбирает СЛУЧАЙНЫЙ включённый маршрут
```

- Точки сортируются по числу в имени (`1`, `2`, `Point3` — главное чтобы была цифра).
- Сервер сам делает точки невидимыми и некликабельными.

### 7.2 Включение/выключение маршрутов

На папке маршрута атрибут **`Enabled`** (bool). Нет атрибута = включён.
`Enabled = false` — враги этот маршрут не выбирают.

### 7.3 События карты между волнами

В `GameManager.server.luau`, таблица `WAVES`:

```lua
{count = 8, spawnDelay = 1, pauseAfter = 10,
 mapEvent = {open = {"Route2"}, close = {"Route1"}}},
```

После окончания этой волны `Route1` закроется, `Route2` откроется, и всем клиентам
прилетит `ReplicatedStorage.MapChanged` (RemoteEvent) с этим событием —
**это точка подключения кат-сцен** (задача 2.1): клиентский скрипт слушает
`MapChanged.OnClientEvent` и показывает камеру/анимацию.

### 7.4 Настройка волн

```lua
local WAVES = {
    {count = 3, spawnDelay = 2, pauseAfter = 5},
    -- count      — сколько врагов
    -- spawnDelay — пауза между спавнами (сек)
    -- pauseAfter — пауза после волны (сек)
    -- hpMult     — множитель HP врагов волны (1.5 = +50%)
    -- speedMult  — множитель скорости
    -- stunners   — сколько врагов волны умеют глушить башни
    -- bosses     — список боссов волны: {"StunCopter", "DestroyerCopter"}
    -- mapEvent   — опционально, см. выше
}
```

## 8. Зоны размещения башен

1. Создай в Workspace папку **`PlacementZones`**.
2. Накидай в неё плоских Part'ов поверх площадок, где башни ставить МОЖНО
   (можно сделать прозрачными).
3. Башня вне всех зон **не активируется** (в Output будет warning).
4. Нет папки — ограничений нет.

Проверка точки: `PlacementZones.isAllowed(position)` из
`ReplicatedStorage.Classes.PlacementZones` — модуль общий, его же будет
использовать клиентский UI расстановки (подсветка можно/нельзя).

## 9. GameData, валюта и сетевые объекты

`GameManager` создаёт в `ReplicatedStorage` папку `GameData`:

| Value | Тип | Что хранит |
|---|---|---|
| `BaseHP` | NumberValue | текущее HP базы |
| `MaxBaseHP` | NumberValue | максимум HP |
| `CurrentWave` | NumberValue | номер текущей волны |
| `MaxWaves` | NumberValue | всего волн |
| `EnemyCount` | NumberValue | живых врагов сейчас |
| `GameOver` | BoolValue | база уничтожена |
| `Victory` | BoolValue | все волны отбиты |

Также в `ReplicatedStorage` живут:

| Объект | Тип | Назначение |
|---|---|---|
| `PlaceTower` | RemoteFunction | клиент просит поставить башню `(towerType, position)` → `{ok, err}` |
| `SellTower` | RemoteFunction | продать свою башню `(towerModel)` → `{ok, refund, err}`; возвращает долю ВСЕХ вложений (покупка + апгрейды) |
| `UpgradeTower` | RemoteFunction | апгрейд башни на уровень `(towerModel)` → `{ok, err}` |
| `RepairTower` | RemoteFunction | мгновенный ремонт `(towerModel)` → `{ok, err}`; отклоняется, если башня «под огнём» |
| `SwitchTowerMode` | RemoteEvent | клиент просит сменить режим башни `(towerModel)` |
| `RequestRestart` | RemoteEvent | «Играть снова» после победы/поражения (принимается только когда игра окончена) |
| `BossWarning` | RemoteEvent | сервер → клиенты: «⚠ приближается босс» (массив имён), за 4 с до спавна |
| `CombatFX` | RemoteEvent | сервер → клиенты: «нарисуй эффект/цифру урона» |
| `MapChanged` | RemoteEvent | сервер сообщает о смене карты (хук кат-сцен) |
| `TowerTemplates` | Folder | модели башен для призраков и спавна |

Атрибуты на моделях башен: `TowerType`, `TowerActive`, `TowerMode`,
`TowerModeName`, `TowerLevel` (уровень прокачки), `OwnerId` (0 = башня с карты),
`OwnerName`, `TowerHealth` / `TowerMaxHealth` (HP),
`TowerAmmo` / `TowerReloading` (магазин), `TowerStunned` (оглушена),
`TowerUnderAttack` (недавно получала урон — ремонт заблокирован),
`TowerDamageDealt` (нанесённый урон — приоритет боссов).
Атрибуты врагов: `Stunner` (стан-враг), `Friendly` (перевербован).
Атрибут боссов: `Boss`.

**Папки в Workspace:** `PlacementZones` (зоны размещения), `Buildings`
(блокеры обзора башен: ЛЮБОЙ Part в папке делает врага за ним невидимым,
рейкаст в `TowerClass:hasLineOfSight`), `WaypointsFolder` (маршруты).

**Тестовая локация:** `src/server/TestMap.luau` при старте достраивает только
маршруты — Route1/Route2 из плоских точек, если папок маршрутов ещё нет.
Выключить: `TestMap.ENABLED = false`.
⚠️ Зоны размещения (PlacementZones) и здания-блокеры (Buildings) НЕ
генерируются — их расставляет картодел руками: создай в Workspace папку
`Buildings` и клади в неё Part'ы там, где башни не должны «видеть» врагов
(например, между маршрутами). Нет папки = обзор ничем не ограничен.

Поля волны в `WAVES`: `count`, `spawnDelay`, `pauseAfter`, `hpMult`, `speedMult`,
`mapEvent`. Волна заканчивается, когда убиты все её враги.

**Валюта:** `player.leaderstats.Money` (IntValue). Старт — 400, награда +25 каждому
игроку за врага, убитого башней (самоподрыв врага о базу награды не даёт),
+100 за босса. Все цены и награды — константы в `Economy.luau` и конфигах башен.

**Сохранение денег:** кошелёк игрока хранится в DataStore `PlayerMoney_v1`
(загрузка при входе, запись при выходе, автосейв раз в минуту, BindToClose).
Рестарт раунда деньги НЕ сбрасывает. ⚠️ В Studio нужно включить
**Game Settings → Security → Enable Studio Access to API Services**,
иначе сохранение молча не работает (все стартуют с 400).

**Рестарт раунда** («Играть снова» после победы/поражения) — «чистое поле»:
враги и боссы убираются, база чинится, КУПЛЕННЫЕ башни удаляются с возвратом
владельцу 100% вложений (если он ещё на сервере), студийные башни чинятся,
а уничтоженные боссом — восстанавливаются из снапшота, который TowerService
делает при первой активации.

⚠️ Анти-эксплойт (задача 5): деньги, постановка и валидация — только на сервере.
Клиент шлёт лишь намерения; сервер перепроверяет тип, цену, зону и дистанцию.

## 9.1 Игровой процесс для игрока

- **Магазин** — панель внизу: клик по кнопке → за курсором ходит призрак башни
  с кругом радиуса (зелёный = можно, красный = нельзя). ЛКМ — поставить, ПКМ/Esc — отмена.
- **Меню башни** — навёл курсор на башню → справа внизу панель: статы, режим,
  круг радиуса; у многорежимных башен кнопка «⇄ Сменить режим».
- **Циферки урона** — всплывают над целью, цвет = тип оружия.

`AdminUI.client.luau` подписан на GameData. Новые элементы UI делаем так же:
сервер пишет в Value, клиент слушает `.Changed`.

## 11. Архитектурные соглашения

- **Источник правды — файлы.** Скрипты в Studio не редактируем.
- Классы — таблицы с `__index`-метатаблицей, конструктор `Class.new(...)`.
- Связь объектов — колбэки: `enemy.onDied`, `base.onDestroyed`.
- Сервер — вся игровая логика; клиент — только UI и (в будущем) кат-сцены.
  Никакого урона/денег на клиенте — это база анти-эксплойта (задача 5).
- Баланс (урон, HP, скорости) — только в конфиг-таблицах вверху файлов
  (`TOWER_CONFIGS`, `CONFIG`, `WAVES`), не в логике.

## 12. Лобби (отдельный place + телепорт)

Лобби — ВТОРОЙ place в том же experience. Игроки жмут «✓ ГОТОВ»;
когда готовы все (минимум `MIN_PLAYERS`), идёт отсчёт и вся компания
телепортируется в игровой place. Код лобби лежит в `src/lobby/` и
синкается СВОИМ Rojo-проектом — `lobby.project.json`.

### 12.1 Настройка с нуля

1. **Создай place**: [Creator Hub](https://create.roblox.com) → твой
   experience → Places → Create Place → назови «Lobby».
2. **Узнай PlaceId игрового плейса**: Creator Hub → Places → у основного
   плейса скопируй ID (или в Studio: File → Game Settings → Basic Info).
3. **Впиши ID**: в `src/lobby/server/LobbyMain.server.luau` поставь
   `GAME_PLACE_ID = <этот ID>`.
4. **Синканись**: открой лобби-плейс в Studio и запусти
   `rojo serve lobby.project.json` (НЕ default!), потом Plugins → Rojo → Connect.
5. **Построй площадку**: спавн, декорации — на вкус. Весь UI лобби
   создаётся кодом, геометрия не обязательна.
6. **Сделай лобби стартовым**: Creator Hub → Places → Lobby → ⋯ →
   Make Start Place, чтобы игроки заходили именно в лобби.

### 12.2 Как это работает

| Кусок | Что делает |
|---|---|
| `LobbyMain.server.luau` | готовность игроков, отсчёт `COUNTDOWN` сек, `TeleportService:TeleportAsync` |
| `LobbyUI.client.luau` | кнопка «✓ ГОТОВ», счётчик «ГОТОВЫ N/M», большой отсчёт |
| `ReplicatedStorage.LobbyData` | `ReadyCount` / `TotalCount` / `Countdown` — Value для UI |
| `ToggleReady` (RemoteEvent) | клиент переключает свою готовность |

Игровой плейс со своей стороны ждёт первого игрока и даёт `START_DELAY`
(10 с) на застройку перед первой волной — см. низ `GameManager.server.luau`.

⚠️ **TeleportService не работает в Studio** — телепорт проверяется только
в опубликованной игре. В Studio лобби напишет warn и никого не отправит.
Сам игровой плейс по-прежнему можно тестировать напрямую — волны стартуют
как только в нём появляется игрок.
